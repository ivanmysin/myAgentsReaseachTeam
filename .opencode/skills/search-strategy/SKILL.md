# Purpose

Спланировать и выполнить многоканальный поиск статей: семантический (ChromaDB + SPECTER2), полнотекстовый (SQL), кластерный (CSV/HTML), и скомбинировать результаты в ранжированный список с обоснованием релевантности.

# When to use

Активируй этот навык когда:
- Нужно найти статьи по теме (любой агент, любая задача)
- Пользователь просит «найди статьи о...», «поищи литературу по...»
- Это начальный этап любого более крупного skills (literature-review, paper-introduction, etc.)
- Нужно оценить объем доступной литературы по теме

# Inputs needed

- Поисковый запрос / тема (на английском для семантического поиска)
- Критерии отбора: временной диапазон, типы статей, журналы
- Ожидаемое количество результатов

# Database structure

SQLite база данных. Путь задан в переменной окружения LITBASE_SQLITE_PATH (читай из файла .env).

Таблица "Articles", колонки:
- id (INTEGER PRIMARY KEY)
- doi (TEXT)
- title (TEXT)
- full_text (TEXT) — полный текст статьи (~15K токенов в среднем, ~10% статей без full_text)
- abstract (TEXT)
- date (TEXT)
- journal (TEXT)
- authors (TEXT)
- volume (TEXT)
- issue (TEXT)
- pages (TEXT)
- pmid (TEXT)
- filepath (TEXT) — путь к исходному PDF

 ChromaDB с эмбеддингами. Путь: LITBASE_CHROMA_PATH. Две коллекции:

- LITBASE_CHROMA_COLLECTION (по умолчанию): абстракты статей (модель SPECTER2)
- CHUNK_CHROMA_COLLECTION: чанки полных текстов (~500 токенов каждый)

CSV файл кластеризации. Путь: LITBASE_CLUSTERS_CSV. Колонки: id, title, doi, date, cluster, similarity.
Примечание: ключевые слова кластеров хранятся в отдельном HTML файле (LITBASE_CLUSTERS_MD).

PDF файлы: LITBASE_PDF_DIR.

# tools

Все инструменты — TypeScript-обертки над Python-скриптами в .opencode/tools/scripts/. Они читают пути из .env автоматически и вызываются как tools по имени.

1. db_search <mode> <query> [options]
   Режимы:
   - keyword <query> — полнотекстовый поиск по title, abstract, full_text
   - sql <SQL-query> — произвольный SQL-запрос к таблице Articles
   - author <name> — поиск по автору
   - doi <doi> — поиск по DOI
   - id <id> — получить статью по ID
   Опции (именованные аргументы):
   - fields: "title,abstract,date" — какие поля возвращать (по умолчанию: id,title,authors,date,doi,abstract)
   - limit: N — максимум результатов (по умолчанию: 20)
   - full: true — включить full_text в результат

 2. semantic_search <query> [options]
    Семантический поиск через ChromaDB. Поддерживает две коллекции:
    - abstracts: абстракты статей (по умолчанию)
    - chunks: чанки полных текстов (~500 токенов)
    - all: обе коллекции одновременно
    Опции (именованные аргументы):
    - top: N — количество результатов (по умолчанию: 20)
    - threshold: FLOAT — минимальная близость (по умолчанию: -1.0)
    - cluster: N — ограничить поиск кластером N
    - collection: "abstracts"|"chunks"|"all" — какую коллекцию искать (по умолчанию: abstracts)

3. read_article <id> [options]
   Чтение полного текста статьи.
   Опции (именованные аргументы):
   - section: "intro"|"methods"|"results"|"discussion"|"all" — попытка извлечь секцию (эвристически)
   - chunk: N — разбить на чанки и вернуть чанк номер N (0-indexed)
   - chunks: true — показать количество чанков
   - chunkSize: N — слов в чанке (по умолчанию: 3000)
   - summary: true — краткое содержание (метаданные + первые 500 слов)
   - full: true — полный текст (ОСТОРОЖНО: ~15K токенов!)

4. extract_figures <id> [options]
   Извлечение рисунков из PDF файла статьи. Сохраняет в указанную директорию.
   Опции (именованные аргументы):
   - output: "path/to/dir"
   - minSize: N — минимальный размер изображения в пикселях (по умолчанию: 100)

5. cluster_browse <mode> [argument]
   Режимы:
   - mode: "list" — показать все кластеры с ключевыми словами и размерами
   - mode: "show", argument: "N" — показать статьи кластера N
   - mode: "find", argument: "query" — найти кластеры по ключевым словам


# Procedure

**Шаг 1 — Анализ запроса:**
1. Выдели ключевые концепции (2-5)
2. Для каждой концепции — список синонимов и связанных терминов
3. Определи, какие поисковые каналы наиболее релевантны:
   - Семантический поиск — для концептуального сходства
   - Ключевые слова — для точных терминов
   - Кластеры — для тематического обзора
   - SQL — для сложных фильтров (дата, журнал, автор)

**Шаг 2 — Раздельный поиск по абстрактам и полным текстам:**
1. Выполни **семантический поиск только по индексу абстрактов** с основным запросом (`--top 30`).
2. Выполни **семантический поиск только по индексу чанков полных текстов** с тем же запросом (`--top 30`).  
   Если запрос длинный, разбей его на 1–2 короткие смысловые части (например, только «метод + результат») и повтори для каждой.
3. Объедини оба списка, удали дубликаты по ID публикации.

**Шаг 3 — Псевдо-релевантная обратная связь:**
1. Из результатов **первого объединённого пула** (Шаг 2) возьми `top‑10` по наибольшему скору.
2. Для этих 10 документов извлеки **5–10 ключевых терминов и фраз**, которые характерны для данной темы, но отсутствовали в исходном запросе.
3. Сформируй **обогащённый запрос**, объединив исходный текст и извлечённые термины.
4. Выполни дополнительный раунд поиска:
   - `semantic_search` по абстрактам с обогащённым запросом (`--top 20`)
   - `semantic_search` по чанкам полных текстов с обогащённым запросом (`--top 20`)
   - `db_search` (keyword) по извлечённым терминам (`--limit 20`)
5. Добавь новые результаты в общий пул, удалив дубликаты.

**Шаг 4 — Поиск по сходству на лучшие документы:**
1. Из пула (после Шага 3) выбери `top‑5` наиболее релевантных публикаций.
2. Для каждой из них выполни **поиск похожих документов**:
   - Если доступен вызов `semantic_search` по ID документа (поиск ближайших соседей по вектору абстракта), используй его (`--top 10`).
   - Иначе — склей заголовок и абстракт документа в текстовый запрос и выполни обычный `semantic_search` по абстрактам (`--top 10`).
3. Добавь найденные публикации в пул, исключив дубликаты.

**Шаг 5 — Декомпозиция сложного запроса и пересечение концепций:**
1. Вернись к анализу запроса из **Шага 1**. Если запрос содержит несколько чётко различимых аспектов (например, «вещество X», «состояние Y», «популяция Z»), разбей его на **атомарные концепции**.
2. Для каждой концепции независимо выполни:
   - `semantic_search` по абстрактам с простой формулировкой этой концепции (`--top 50`)
   - `db_search` (keyword) по ключевым терминам этой концепции (`--limit 50`)
3. Собери множества ID публикаций для каждой концепции.
4. **Найди пересечение** этих множеств — только публикации, в которых присутствуют все аспекты.
   - Если ID хранятся в SQL-базе, используй `INTERSECT` трёх выборок.
   - Иначе — программное пересечение списков ID.
5. Добавь публикации из пересечения в общий пул.

**Шаг 7 — Поиск по ключевым авторам:**
1. Из текущего объединённого пула (после всех предыдущих шагов) извлеки всех авторов.  
   Определи **наиболее часто встречающихся** (например, имеющих ≥2 публикаций в релевантном пуле).
2. Для каждого такого автора выполни **адресный поиск**:
   - `db_search` (keyword) с фамилией и инициалами (если поддерживается полнотекстовый поиск по автору);
   - или SQL‑запрос вида `SELECT * FROM papers WHERE LOWER(author) LIKE '%фамилия_и.%'` с ограничением по годам (например, последние 10 лет).
3. Добавь все найденные работы в общий пул.

**Шаг 8 —Финальное объединение и переранжирование:**
После выполнения всех шагов собери уникальные ID из всех источников, удали дубликаты и примени взвешенное переранжирование на основе:
- скоров из семантического поиска,
- частоты попадания в разные каналы,
- позиций в ранжированных списках.

**Шаг 9 — Сохранение:**
1. Сохрани результаты в `output/search_log_<task>.md`
2. Включи: запросы, количество результатов по каналам, финальный ranked list

# Quality bar (self-check)

- [ ] Использовано минимум 2 поисковых канала (не только один)
- [ ] Поисковые запросы на АНГЛИЙСКОМ языке
- [ ] Вариации запроса покрывают синонимы
- [ ] Удалены дубликаты
- [ ] Для каждой статьи — краткое обоснование релевантности
- [ ] Full text НЕ загружается (только метаданные и абстракты)
- [ ] Результаты сохранены в файл
- [ ] При большом количестве — группировка по подтемам

# Anti-patterns

- ❌ Использовать только semantic_search без keyword
- ❌ Использовать только keyword без semantic — упускаешь концептуально близкие статьи
- ❌ Делать один запрос и останавливаться — всегда пробуй вариации
- ❌ Загружать full_text на этапе поиска (это задача @reader)
- ❌ Пропускать кластерную навигацию для больших тем
- ❌ Возвращать unsorted list без ранжирования и обоснования
- ❌ Формулировать поисковые запросы на русском (статьи в базе на английском)

# Examples

**Input:** «Найди статьи о взаимодействии тета-осцилляций и синаптической пластичности в гиппокампе»
**Output:** 4 канала поиска: (1) semantic "theta oscillations synaptic plasticity hippocampus", (2) semantic "theta phase LTP LTD STDP", (3) keyword "theta AND plasticity AND hippocampus", (4) cluster find "plasticity theta". Объединено 85 результатов → 23 уникальных → топ-15 с обоснованием релевантности.
